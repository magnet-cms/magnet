# Admin UI Internationalization (i18n) Implementation Plan

Created: 2026-03-14
Status: COMPLETE
Approved: Yes
Iterations: 0
Worktree: No
Type: Feature

## Summary

**Goal:** Extract all hardcoded English strings from the Admin UI into a message catalog system using `react-intl` (FormatJS), generate message files for all existing pages and components, and provide a locale switcher in the Admin UI General Settings section.

**Architecture:** Hook-first approach using `useIntl()` + `intl.formatMessage()` throughout. Manually maintained `en.json` catalog as source of truth (no @formatjs/cli extraction pipeline). `AppIntlProvider` wraps the entire app as the **outermost** provider (does NOT depend on MagnetProvider), with `locale` and `messages` props exposed on the `MagnetAdmin` library component. Locale persisted in localStorage with browser language fallback. Non-English locales lazy-loaded via dynamic `import()`.

**Provider stack (outermost ŌåÆ innermost):** `AppIntlProvider` ŌåÆ `MagnetProvider` ŌåÆ `QueryClientProvider` ŌåÆ `DialogProvider` ŌåÆ `Router`

**Tech Stack:** `react-intl` (FormatJS), ICU message format, dynamic imports for locale bundles.

## Scope

### In Scope
- Install `react-intl` dependency
- Create i18n module: IntlProvider wrapper, typed `useAppIntl()` hook, message types
- Create `en.json` message catalog with all extracted strings
- Generate `pt-BR.json` and `es.json` stub files
- Replace all hardcoded strings in all 95 TSX files with `intl.formatMessage()` calls
- Internationalize Zod validation schemas (login, signup, profile-setup forms)
- Replace hardcoded `toLocaleDateString('en-US')` with `intl.formatDate()` and `intl.formatNumber()`
- Add `locale` and `messages` props to `MagnetAdmin` component
- Add locale switcher in General Settings section
- Lazy-load non-English locales to avoid bundle bloat
- Update MDX documentation

### Out of Scope
- ESLint/Biome lint rule for catching hardcoded strings (Biome doesn't support formatjs plugin; rely on code review)
- @formatjs/cli extraction/compilation pipeline (manual catalog approach chosen)
- Machine translation of pt-BR/es catalogs (stubs with English strings only)
- Backend API response internationalization
- UI component library (`@magnet-cms/ui`) string internationalization
- Plugin frontend internationalization (content-builder, SEO, vault ŌĆö each plugin handles its own i18n)
- Turbo pipeline tasks for intl:extract/compile (not needed with manual approach)

## Context for Implementer

> Write for an implementer who has never seen the codebase.

- **Patterns to follow:**
  - Hook-based React patterns throughout: see `packages/client/admin/src/hooks/` for existing custom hooks
  - Context/Provider pattern: see `packages/client/admin/src/core/provider/MagnetProvider.tsx` for how providers wrap the app
  - Feature-based organization: `src/features/{feature}/components/` ŌĆö each feature is self-contained
  - Toast messages use `sonner`: `import { toast } from 'sonner'` with `toast.success()` / `toast.error()`

- **Conventions:**
  - Message ID format: `{feature}.{section}.{element}` ŌĆö e.g., `auth.login.title`, `dashboard.stats.collections`, `common.actions.save`
  - Use `intl.formatMessage({ id: 'key', defaultMessage: 'English text' })` for all strings
  - Default messages must always be provided (serves as documentation and fallback)
  - File imports use `~/*` alias for `src/` directory
  - Biome formatting: single quotes, no semicolons, 2-space indent, tab indent for JSX

- **Key files:**
  - `src/App.tsx` ŌĆö dev mode entry point (BrowserRouter)
  - `src/MagnetAdmin.tsx` ŌĆö library entry point (configurable)
  - `src/core/provider/MagnetProvider.tsx` ŌĆö main context provider
  - `src/core/dialog/DialogService.tsx` ŌĆö confirmation dialogs
  - `src/layouts/DashboardLayout/AppSidebar.tsx` ŌĆö sidebar navigation with hardcoded strings
  - `src/layouts/DashboardLayout/NavUser.tsx` ŌĆö user dropdown menu
  - `src/features/settings/components/SettingsPage.tsx` ŌĆö settings page with dynamic tabs
  - `src/features/settings/components/DynamicSettingsForm.tsx` ŌĆö dynamic form for backend settings

- **Gotchas:**
  - The admin is both a standalone Vite app AND a library package. Changes must work in both modes.
  - `MagnetAdmin` is the public API ŌĆö any new prop is a breaking change if not optional.
  - Settings page tabs are dynamically loaded from backend schemas. The locale switcher is a client-side-only concern, not a backend setting. It should be rendered as a separate card within the General settings tab.
  - The existing `components/LocaleSwitcher/` is for content locale switching (draft/published), NOT UI language switching. Don't confuse the two.
  - Zod schemas are defined at module scope. To internationalize them, you need to create factory functions that accept an `IntlShape` instance and return the schema.
  - Plugin frontends (vault, content-builder, SEO) have their own TSX files ŌĆö those are out of scope for this plan.

- **Domain context:**
  - "Content Manager" is the main CMS feature ŌĆö CRUD for dynamic schemas
  - "Schema" refers to database model definitions discovered at runtime
  - "Settings" are key-value pairs grouped by backend modules using `@Settings` decorator

## Assumptions

- `react-intl` version ^7.x is compatible with React 18/19 ŌĆö supported by [react-intl docs](https://formatjs.github.io/docs/react-intl). Tasks 1-12 depend on this.
- Dynamic `import()` for locale JSON files works with Vite's code splitting ŌĆö supported by Vite natively. Task 2 depends on this.
- The General settings tab has id "general" in the frontend ŌĆö supported by `general.settings.ts` setting `group: 'general'` (line 38) and `parseSettingsTabs()` using `schema.settingsOptions?.group` as tab id (parseSchema.ts:20). Task 11 depends on this.
- Toast messages from `sonner` accept translated strings ŌĆö they accept any string. Tasks 4-10 depend on this.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Bundle size increase from react-intl | Medium | Low | react-intl is ~15KB gzipped. Locale files lazy-loaded. Monitor with `bun run analyze`. |
| Message ID typos cause runtime fallback to defaultMessage | Medium | Low | TypeScript union type for message IDs catches typos at compile time via `useAppIntl()` hook. |
| Zod schema factory functions break form validation | Low | High | Test each form (login, signup, profile-setup) after conversion. Zod schemas must still validate identically. |
| Plugin frontends break when IntlProvider is added | Low | Medium | IntlProvider is added above the plugin mount point. Plugins don't use intl unless they opt in. Test with content-builder plugin loaded. |
| Library consumers pass invalid locale | Low | Low | Fallback chain: prop locale ŌåÆ localStorage ŌåÆ navigator.language ŌåÆ 'en'. Always falls back to English. |

**Objective:** Create intl-aware Zod schema factory functions so validation messages are translated.

**Dependencies:** Task 2

- Modify: `src/i18n/types.ts`

**Key Decisions / Notes:**
- Current pattern: `z.string().min(1, 'Email is required')` ŌĆö hardcoded at module scope
- New pattern: Move schema creation inside the component (or a custom hook) where `intl` is available:
  ```tsx
  const intl = useAppIntl()
  const loginSchema = useMemo(() => z.object({
    email: z.string().min(1, intl.formatMessage({ id: 'validation.emailRequired', defaultMessage: 'Email is required' })),
    ...
  }), [intl])
  ```
- Use `useMemo` with `intl` dependency to avoid recreating on every render (only when locale changes)
- Validation message IDs: `validation.required` ("{field} is required"), `validation.email` ("Invalid email address"), `validation.minLength` ("Must be at least {min} characters"), etc.
- Password rules: `validation.password.lowercase` ("Must contain at least one lowercase letter"), etc.
- **Stale schema edge case:** `react-hook-form` uses `zodResolver(schema)` which captures the schema at creation time. When `intl` changes (locale switch), the `useMemo` returns a new schema, but the resolver must also be recreated. Pass the `resolver` as part of the `useMemo` dependency chain, or accept that validation messages only update on component remount (acceptable for this project since locale changes trigger a full re-render from the root IntlProvider).
