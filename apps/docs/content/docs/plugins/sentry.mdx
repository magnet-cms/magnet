The Sentry plugin integrates [Sentry](https://sentry.io) into your Magnet CMS application, providing error tracking, performance monitoring, scheduled job monitoring, and a user feedback widget in the admin UI.

## Installation

```bash
bun add @magnet-cms/plugin-sentry @sentry/nestjs
```

## Quick Setup

```typescript
import { MagnetModule } from '@magnet-cms/core'
import { SentryPlugin } from '@magnet-cms/plugin-sentry'
import { Module } from '@nestjs/common'

@Module({
  imports: [
    MagnetModule.forRoot([
      SentryPlugin.forRoot({
        dsn: process.env.SENTRY_DSN,
        tracesSampleRate: 0.1,
        environment: process.env.NODE_ENV,
      }),
    ]),
  ],
})
export class AppModule {}
```

```bash
# .env
SENTRY_DSN=https://your-dsn@o0.ingest.sentry.io/12345
```

The DSN is also auto-resolved from the `SENTRY_DSN` environment variable when not provided to `forRoot()`.

## Configuration

All options are optional — the plugin works with just a DSN.

```typescript
SentryPlugin.forRoot({
  // Sentry Data Source Name — required for any events to be sent
  dsn: process.env.SENTRY_DSN,

  // Sample rate for performance tracing (0.0 to 1.0)
  // Use 1.0 in development, lower values (0.1) in production
  tracesSampleRate: 0.1,

  // Deployment environment tag
  environment: process.env.NODE_ENV ?? 'production',

  // Release version — tag events with the current deploy
  release: process.env.SENTRY_RELEASE,

  // Enable Sentry debug logging (not recommended in production)
  debug: false,

  // Set to false to disable all Sentry activity (e.g., in test environments)
  enabled: process.env.NODE_ENV !== 'test',

  // Attach stack traces to all events, not just errors
  attachStacktrace: true,

  // Maximum breadcrumbs per event
  maxBreadcrumbs: 100,
})
```

## Error Tracking

Exceptions thrown in any controller are automatically captured and sent to Sentry. The plugin enriches each event with:

- **User context** — `userId` and `ipAddress` from the authenticated session
- **Request ID** — the `X-Request-ID` tag for log correlation
- **Breadcrumbs** — automatic breadcrumb trail

No additional setup required — error capture is active as soon as the plugin is registered.

## Performance Monitoring

### Automatic HTTP Tracing

When `tracesSampleRate > 0`, `@sentry/nestjs` automatically instruments incoming HTTP requests as Sentry transactions. No additional setup needed.

### Recommended: Early Initialization

For full performance tracing (including NestJS module initialization, database connections, and other startup work), initialize Sentry before NestJS boots by creating an `instrument.ts` file:

```typescript
// src/instrument.ts
import { initSentryInstrumentation } from '@magnet-cms/plugin-sentry'

initSentryInstrumentation({
  dsn: process.env.SENTRY_DSN,
  tracesSampleRate: 1.0,
})
```

Then import it as the very first line of your `main.ts`:

```typescript
// src/main.ts
import './instrument'
import { NestFactory } from '@nestjs/core'
import { AppModule } from './app.module'

async function bootstrap() {
  const app = await NestFactory.create(AppModule)
  await app.listen(3000)
}
bootstrap()
```

> **Note:** When using `instrument.ts`, the plugin's `onPluginInit()` hook automatically detects that Sentry is already initialized and skips the duplicate `Sentry.init()` call.

### Custom Spans

Use the `@SentrySpan` decorator or `withSentrySpan` helper to instrument specific operations:

```typescript
import { Injectable } from '@nestjs/common'
import { SentrySpan, withSentrySpan } from '@magnet-cms/plugin-sentry'

@Injectable()
export class ReportService {
  // Decorator approach — wraps the async method in a Sentry span
  @SentrySpan('generate-report', 'task')
  async generateReport(id: string): Promise<Report> {
    return this.buildReport(id)
  }

  // Programmatic approach — useful for conditional spans
  async processData(items: string[]) {
    return withSentrySpan('process-data', 'function', async () => {
      for (const item of items) {
        await this.processItem(item)
      }
    })
  }
}
```

Both helpers gracefully degrade to no-ops when Sentry is not initialized.

## Cron Monitoring

Monitor scheduled tasks using the `@SentryCron` or `@MagnetSentryCron` decorators. Apply them **after** `@Cron()` from `@nestjs/schedule`:

```typescript
import { Injectable } from '@nestjs/common'
import { Cron, CronExpression } from '@nestjs/schedule'
import { SentryCron, MagnetSentryCron } from '@magnet-cms/plugin-sentry'

@Injectable()
export class TasksService {
  // Explicit monitor slug
  @Cron(CronExpression.EVERY_HOUR)
  @SentryCron('hourly-digest', {
    schedule: { type: 'crontab', value: '0 * * * *' },
    checkinMargin: 5,
    maxRuntime: 30,
  })
  async sendDigest() {
    // Sentry alerts if this job misses its schedule or runs too long
  }

  // Auto-generated slug from class + method name: "tasksservice-cleanup"
  @Cron(CronExpression.EVERY_DAY_AT_MIDNIGHT)
  @MagnetSentryCron()
  async cleanup() {
    // Slug auto-generated as "tasksservice-cleanup"
  }
}
```

`@MagnetSentryCron()` auto-generates the monitor slug from `ClassName-methodName` (lowercased, special chars replaced with `-`). Pass an explicit slug as the first argument to override.

## User Feedback Widget

When the plugin's frontend bundle is loaded in the admin UI, a floating "Report an Issue" button appears that opens Sentry's feedback dialog. Users can describe issues and optionally attach screenshots.

The widget is powered by `@sentry/browser` and uses shadow DOM for style isolation.

## Environment Variables

| Variable | Description | Required |
|----------|-------------|----------|
| `SENTRY_DSN` | Sentry Data Source Name | Yes (or pass via `forRoot`) |
| `SENTRY_RELEASE` | Release version tag | No |
| `SENTRY_ENVIRONMENT` | Environment override | No (falls back to `NODE_ENV`) |
