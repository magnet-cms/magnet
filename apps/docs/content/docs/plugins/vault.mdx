## Overview

The Vault plugin integrates [HashiCorp Vault](https://www.vaultproject.io/) with Magnet CMS for centralized secrets management. It supports both remote Vault servers and local development with environment variable fallback.

**Key features:**
- Boot-time secret resolution via `VaultBootstrap.resolve()`
- Runtime secret resolution via `VaultService` with caching and rotation
- Convention-based path mapping with explicit overrides
- Admin UI for connection config, secret browsing, and mapping management
- Token and AppRole authentication methods

## Installation

```bash
bun add @magnet-cms/plugin-vault
```

## Quick Start

### 1. Register the plugin

```typescript
import { MagnetModule } from '@magnet-cms/core'
import { VaultPlugin } from '@magnet-cms/plugin-vault'

@Module({
  imports: [
    MagnetModule.forRoot({
      db: { uri: process.env.MONGODB_URI || 'mongodb://localhost:27017/myapp' },
      jwt: { secret: process.env.JWT_SECRET || 'dev-secret' },
      plugins: [{ plugin: VaultPlugin }],
    }),
  ],
})
export class AppModule {}
```

### 2. Boot-time secrets (optional)

For production, use `VaultBootstrap.resolve()` to fetch secrets from Vault before NestJS boots:

```typescript
import { MagnetModule } from '@magnet-cms/core'
import { VaultBootstrap, VaultPlugin } from '@magnet-cms/plugin-vault'

// Resolve secrets before module initialization
const secrets = await VaultBootstrap.resolve({
  fallbackToEnv: true,
  paths: {
    db: { vaultPath: 'secret/data/magnet/database', envPrefix: 'DB_' },
    jwt: { vaultPath: 'secret/data/magnet/jwt', envPrefix: 'JWT_' },
  },
})

@Module({
  imports: [
    MagnetModule.forRoot({
      db: { uri: secrets.db.uri as string },
      jwt: { secret: secrets.jwt.secret as string },
      plugins: [{ plugin: VaultPlugin }],
    }),
  ],
})
export class AppModule {}
```

## Configuration

### Environment Variables

| Variable | Description |
|----------|-------------|
| `VAULT_ADDR` | Vault server URL (e.g., `https://vault.example.com:8200`) |
| `VAULT_TOKEN` | Vault authentication token |
| `VAULT_ROLE_ID` | AppRole role ID (when using AppRole auth) |
| `VAULT_SECRET_ID` | AppRole secret ID (when using AppRole auth) |

### Authentication Methods

**Token authentication** (simplest):
```bash
export VAULT_ADDR=https://vault.example.com:8200
export VAULT_TOKEN=hvs.your-token-here
```

**AppRole authentication** (recommended for production):
```typescript
const secrets = await VaultBootstrap.resolve({
  url: 'https://vault.example.com:8200',
  auth: {
    type: 'appRole',
    roleId: process.env.VAULT_ROLE_ID!,
    secretId: process.env.VAULT_SECRET_ID,
  },
  paths: {
    db: {},
    jwt: {},
  },
})
```

### Path Mapping

**Convention-based** (zero config):

By default, secrets are read from `secret/data/magnet/{module}`:
- `secret/data/magnet/database` → database config
- `secret/data/magnet/jwt` → JWT config
- `secret/data/magnet/email` → email config

**Explicit overrides:**

Override specific paths when your Vault layout differs:

```typescript
const secrets = await VaultBootstrap.resolve({
  paths: {
    db: { vaultPath: 'secret/data/myapp/prod/database' },
    jwt: { vaultPath: 'kv/shared/jwt-keys' },
    email: {}, // uses convention: secret/data/magnet/email
  },
})
```

## Runtime Usage

Inject `VaultService` for dynamic secret resolution with caching:

```typescript
import { Injectable } from '@nestjs/common'
import { VaultService } from '@magnet-cms/plugin-vault'

@Injectable()
export class MyService {
  constructor(private readonly vault: VaultService) {}

  async getApiKey(): Promise<string | undefined> {
    const secret = await this.vault.get<{ apiKey: string }>('magnet/api-keys')
    return secret?.apiKey
  }
}
```

### Watching for Rotation

For secrets that rotate (API keys, tokens), use `watch()` to periodically refresh:

```typescript
// In your module's onModuleInit
vault.watch('magnet/api-keys', 60_000) // refresh every 60 seconds
```

## Admin UI

The plugin adds a **Vault** entry to the admin sidebar with:

- **Connection Panel** — shows Vault status (connected/disconnected/not configured) and a "Test Connection" button
- **Secret Browser** — tree view of secret paths at the configured mount (key names only, not values)
- **Mappings Panel** — manage explicit path-to-config mappings with add/edit/delete

Vault settings are also available in the Settings panel under the "Vault" tab.

## Local Development

### Environment variable fallback (recommended)

When `VAULT_ADDR` is not set, `VaultBootstrap.resolve()` falls back to reading environment variables:

```bash
# .env file
DB_URI=mongodb://localhost:27017/myapp
DB_HOST=localhost
JWT_SECRET=development-secret-key
EMAIL_HOST=localhost
EMAIL_PORT=1025
```

The env prefix is derived from the module name (`DB_` for `db`, `JWT_` for `jwt`) or set explicitly via `envPrefix`.

### Local Vault dev server

For teams wanting Vault parity in development:

```bash
# Start a dev Vault server
vault server -dev -dev-root-token-id="dev-token"

# Set environment
export VAULT_ADDR=http://127.0.0.1:8200
export VAULT_TOKEN=dev-token

# Seed secrets
vault kv put secret/magnet/database uri=mongodb://localhost:27017/myapp
vault kv put secret/magnet/jwt secret=development-secret-key
```

## Production Setup

1. **Configure Vault access:**
   - Create an AppRole for your Magnet CMS application
   - Grant read access to `secret/data/magnet/*`
   - Set `VAULT_ADDR`, `VAULT_ROLE_ID`, and `VAULT_SECRET_ID`

2. **Store secrets in Vault:**
   ```bash
   vault kv put secret/magnet/database uri=$DATABASE_URL
   vault kv put secret/magnet/jwt secret=$JWT_SECRET
   vault kv put secret/magnet/email host=$SMTP_HOST port=$SMTP_PORT
   ```

3. **Use VaultBootstrap in your app module** to resolve secrets at boot time

4. **Configure admin settings** via the Vault settings page in the admin panel

## API Reference

### VaultBootstrap

| Method | Description |
|--------|-------------|
| `resolve(options)` | Fetch secrets from Vault or env vars before NestJS boots |

### VaultService

| Method | Description |
|--------|-------------|
| `get<T>(path)` | Read a secret with caching |
| `getMany(paths)` | Read multiple secrets |
| `refresh(path)` | Force-refresh a cached secret |
| `watch(path, intervalMs?)` | Start periodic refresh for rotation |
| `unwatch(path)` | Stop watching a secret |
| `isConfigured()` | Check if Vault is connected |
| `healthCheck()` | Test Vault server health |

### REST Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/vault/status` | Connection status |
| POST | `/vault/test-connection` | Test connectivity |
| GET | `/vault/secrets` | List secret paths |
| GET | `/vault/secrets/:path` | Get secret key names |
| GET | `/vault/mappings` | List configured mappings |
| PUT | `/vault/mappings` | Update mappings |
