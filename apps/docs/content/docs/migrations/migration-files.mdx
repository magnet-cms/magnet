# Migration Files

Migration files are TypeScript modules stored in your `migrations/` directory. Each file exports a `Migration` object with `up` and `down` functions.

## File Format

```typescript
// migrations/0001_initial_schema.ts
import type { Migration, MigrationDb } from '@magnet-cms/adapter-drizzle'

export const migration: Migration = {
  id: '0001_initial_schema',
  timestamp: 1710000000000,

  async up(db: MigrationDb) {
    await db.execute(`
      CREATE TABLE "posts" (
        "id" UUID PRIMARY KEY DEFAULT gen_random_uuid(),
        "title" TEXT NOT NULL,
        "body" TEXT,
        "published_at" TIMESTAMP,
        "created_at" TIMESTAMP NOT NULL DEFAULT now()
      )
    `)
  },

  async down(db: MigrationDb) {
    await db.execute(`DROP TABLE "posts"`)
  },
}
```

## The `Migration` Interface

```typescript
interface Migration {
  id: string           // Unique identifier, matches filename prefix
  timestamp: number    // Unix timestamp (milliseconds) when migration was created
  description?: string // Optional human-readable description
  dangerous?: boolean  // True if migration contains destructive operations
  warnings?: string[]  // List of warning messages for dangerous operations

  up(db: MigrationDb): Promise<void>   // Apply the migration
  down(db: MigrationDb): Promise<void> // Revert the migration
}
```

## The `MigrationDb` Interface

The `db` parameter in `up` and `down` is a `MigrationDb` — a minimal interface for raw SQL execution:

```typescript
interface MigrationDb {
  execute(sql: string, params?: unknown[]): Promise<unknown>
}
```

Use it for all DDL and DML in your migrations:

```typescript
async up(db: MigrationDb) {
  // Simple DDL
  await db.execute(`ALTER TABLE "users" ADD COLUMN "phone" TEXT`)

  // With parameters
  await db.execute(
    `INSERT INTO "settings" ("key", "value") VALUES ($1, $2)`,
    ['default_locale', 'en']
  )
}
```

## File Naming

Migration files use a sequential numbering prefix: `NNNN_description.ts`

- `0001_initial_schema.ts`
- `0002_add_user_profiles.ts`
- `0003_add_email_indexes.ts`

The `id` field in the migration object must match the filename prefix (everything before `.ts`).

When generated by `magnet migrate:generate` or `magnet migrate:create`, the number is assigned automatically.

## Multi-Statement Migrations

For multiple SQL statements in a single migration, call `db.execute()` multiple times:

```typescript
async up(db: MigrationDb) {
  await db.execute(`
    CREATE TABLE "tags" (
      "id" UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      "name" TEXT NOT NULL UNIQUE
    )
  `)

  await db.execute(`
    CREATE TABLE "post_tags" (
      "post_id" UUID NOT NULL REFERENCES "posts"("id") ON DELETE CASCADE,
      "tag_id" UUID NOT NULL REFERENCES "tags"("id") ON DELETE CASCADE,
      PRIMARY KEY ("post_id", "tag_id")
    )
  `)

  await db.execute(`
    CREATE INDEX "post_tags_tag_id_idx" ON "post_tags" ("tag_id")
  `)
}
```

## SQL by Dialect

Migration SQL syntax varies by database. The SQL generator helpers in `@magnet-cms/adapter-drizzle` cover the most common operations:

### PostgreSQL

```typescript
import * as pg from '@magnet-cms/adapter-drizzle/sql-generators/postgresql'

async up(db: MigrationDb) {
  // CREATE TABLE
  await db.execute(pg.createTable('products', [
    { name: 'id', type: 'UUID', primaryKey: true },
    { name: 'name', type: 'TEXT', nullable: false },
    { name: 'price', type: 'NUMERIC(10,2)', nullable: false },
  ]))

  // ADD COLUMN
  await db.execute(pg.addColumn('products', { name: 'sku', type: 'TEXT' }))

  // CREATE INDEX
  await db.execute(pg.createIndex({
    name: 'products_sku_idx',
    table: 'products',
    columns: ['sku'],
    unique: true,
  }))
}

async down(db: MigrationDb) {
  await db.execute(pg.dropIndex('products_sku_idx'))
  await db.execute(pg.dropTable('products'))
}
```

### MySQL

```typescript
import * as mysql from '@magnet-cms/adapter-drizzle/sql-generators/mysql'

async up(db: MigrationDb) {
  await db.execute(mysql.createTable('products', [
    { name: 'id', type: 'VARCHAR(36)', primaryKey: true },
    { name: 'name', type: 'VARCHAR(255)', nullable: false },
  ]))

  // MySQL uses MODIFY COLUMN for type changes
  await db.execute(mysql.modifyColumn('products', {
    name: 'name',
    type: 'TEXT',
    nullable: false,
  }))
}
```

### SQLite

```typescript
import * as sqlite from '@magnet-cms/adapter-drizzle/sql-generators/sqlite'

async up(db: MigrationDb) {
  await db.execute(sqlite.createTable('products', [
    { name: 'id', type: 'TEXT', primaryKey: true },
    { name: 'name', type: 'TEXT', nullable: false },
  ]))

  // Note: SQLite ADD COLUMN cannot have NOT NULL without DEFAULT
  // The sqlite generator handles this automatically
  await db.execute(sqlite.addColumn('products', { name: 'sku', type: 'TEXT' }))
}
```

## Dangerous Operations

Some operations are flagged as dangerous because they can cause data loss:

| Operation | Reason |
|-----------|--------|
| `DROP TABLE` | Deletes all data in the table |
| `DROP COLUMN` | Deletes all data in the column |
| `ALTER COLUMN TYPE` | May truncate or fail with existing data |
| `MODIFY COLUMN` (MySQL) | Same as ALTER COLUMN TYPE |

When `magnet migrate:generate` detects these, it:
1. Sets `dangerous: true` on the migration object
2. Populates `warnings[]` with descriptions
3. Displays warnings in the console before generating the file

You can check programmatically:

```typescript
export const migration: Migration = {
  id: '0005_remove_legacy_column',
  timestamp: 1710000000000,
  dangerous: true,
  warnings: ['Drops "users"."legacy_field" — all data will be lost'],

  async up(db: MigrationDb) {
    await db.execute(`ALTER TABLE "users" DROP COLUMN "legacy_field"`)
  },

  async down(db: MigrationDb) {
    await db.execute(`ALTER TABLE "users" ADD COLUMN "legacy_field" TEXT`)
  },
}
```

## Checksum Validation

The migration runner computes a SHA-256 checksum of each migration's `up` and `down` function source when it is first applied. On subsequent runs, checksums are verified. If a migration file has been modified after being applied, the runner throws a `MigrationChecksumError` and stops.

**Never modify applied migration files.** If you need to change behavior, create a new migration.

## Transactions

By default (`transactional: true` in config), each migration's `up` or `down` function is wrapped in a transaction. If the function throws, the database is rolled back to its state before the migration started.

This means partial schema changes from a failed migration are automatically reverted.
