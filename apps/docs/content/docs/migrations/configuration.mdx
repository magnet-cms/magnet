# Migration Configuration

The migration system is configured through the `migrations` property in your `db` config inside `MagnetModule.forRoot()`.

## Full Configuration

```typescript
MagnetModule.forRoot({
  db: {
    connectionString: process.env.DATABASE_URL || '',
    dialect: 'postgresql',
    driver: 'neon',
    migrations: {
      // 'auto': detect + generate + apply on startup (development)
      // 'manual': detect + warn only, no auto-apply (production)
      mode: process.env.NODE_ENV === 'production' ? 'manual' : 'auto',

      // Directory where migration files are stored (relative to project root)
      directory: './migrations',

      // Name of the migrations history table (default: '_magnet_migrations')
      tableName: '_magnet_migrations',

      // Name of the migrations lock table (default: '_magnet_migrations_lock')
      lockTableName: '_magnet_migrations_lock',

      // Lock timeout in milliseconds — stale locks older than this are cleared
      // Default: 60000 (1 minute)
      lockTimeout: 60000,

      // Whether to wrap each migration in a transaction
      // Default: true
      transactional: true,
    },
  },
})
```

## Configuration Options

### `mode`

Controls how Magnet handles schema changes on startup.

| Value | Behavior |
|-------|----------|
| `'auto'` | Detect changes → generate migration file → apply immediately |
| `'manual'` | Detect changes → log warning → do nothing |

Use `'auto'` for local development. Use `'manual'` for staging and production environments.

**Recommended pattern:**

```typescript
migrations: {
  mode: process.env.NODE_ENV === 'production' ? 'manual' : 'auto',
  directory: './migrations',
}
```

### `directory`

The directory where migration files are stored, relative to `process.cwd()` (your project root).

```typescript
// Default
directory: './migrations'

// Absolute path also works
directory: '/app/migrations'
```

Migration files generated by `magnet migrate:generate` and `magnet migrate:create` are written to this directory. **Commit these files to version control.**

### `tableName`

The name of the table used to track applied migrations. Default: `_magnet_migrations`.

Only change this if `_magnet_migrations` conflicts with an existing table name in your schema.

### `lockTableName`

The name of the table used to prevent concurrent migration runs. Default: `_magnet_migrations_lock`.

### `lockTimeout`

How long (in milliseconds) before a lock is considered stale and cleared. Default: `60000` (1 minute).

When a migration process crashes mid-run, it leaves a lock behind. The next migration run will clear locks older than `lockTimeout`. Set this to a value longer than your longest expected migration run time.

### `transactional`

Whether to wrap each individual migration in a database transaction. Default: `true`.

When `true`, a failing migration is automatically rolled back. When `false`, migrations run without transactions — useful for databases that don't support DDL in transactions (some MySQL versions).

## Backward Compatibility

If you omit the `migrations` config entirely, Magnet falls back to its previous behavior: `CREATE TABLE IF NOT EXISTS` for each schema on startup. This preserves compatibility for projects that haven't adopted the migration system yet.

```typescript
// No migrations config = legacy behavior (no migration files)
MagnetModule.forRoot({
  db: {
    connectionString: process.env.DATABASE_URL || '',
    dialect: 'postgresql',
    driver: 'neon',
    // No migrations property = CREATE TABLE IF NOT EXISTS on startup
  },
})
```

## Environment-Based Configuration

A common pattern for multi-environment setups:

```typescript
// config/database.config.ts
export const getDatabaseConfig = () => ({
  connectionString: process.env.DATABASE_URL || '',
  dialect: 'postgresql' as const,
  driver: (process.env.DB_DRIVER || 'neon') as 'neon' | 'pg',
  migrations: process.env.MIGRATIONS_ENABLED === 'false'
    ? undefined
    : {
        mode: (process.env.NODE_ENV === 'production' ? 'manual' : 'auto') as 'auto' | 'manual',
        directory: process.env.MIGRATIONS_DIR || './migrations',
        lockTimeout: parseInt(process.env.MIGRATIONS_LOCK_TIMEOUT || '60000'),
      },
})
```

Then in your app module:

```typescript
// app.module.ts
import { getDatabaseConfig } from './config/database.config'

MagnetModule.forRoot({
  db: getDatabaseConfig(),
  jwt: { secret: process.env.JWT_SECRET || '' },
})
```
