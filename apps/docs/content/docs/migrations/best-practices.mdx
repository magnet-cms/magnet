# Best Practices

## Development vs Production

### Development: Use Auto Mode

In development, `mode: 'auto'` keeps your database in sync with your schema decorators automatically. Every time you start the server with new schema changes, Magnet generates and applies the migration for you.

```typescript
migrations: {
  mode: process.env.NODE_ENV === 'production' ? 'manual' : 'auto',
  directory: './migrations',
}
```

**Commit the generated migration files.** Auto-mode creates real files in your `migrations/` directory. Commit them — they document your schema history and allow teammates to apply the same changes.

### Production: Use Manual Mode

In production, never apply migrations automatically on startup. Use `mode: 'manual'` and deploy migrations as an explicit step:

```bash
# In your deployment pipeline
bun run migrate:status  # Review pending
bun run migrate:up      # Apply
```

This gives you control over when schema changes happen, independent of application deployments.

## Team Workflow

### Schema Change Workflow

1. **Modify schema decorator** — add field, change type, etc.
2. **Start dev server** — auto-mode generates and applies the migration
3. **Review the generated file** — check `migrations/0NNN_auto.ts`
4. **Rename if needed** — rename from `auto` to a descriptive name:
   ```bash
   mv migrations/0003_auto.ts migrations/0003_add_user_display_name.ts
   # Update the id field in the file to match
   ```
5. **Commit** — commit both the schema change and the migration file

### Migration File Names

Give migrations meaningful names that describe what they do:

```
0001_initial_schema.ts         ✓ Clear
0002_add_user_profiles.ts      ✓ Clear
0003_add_email_unique_index.ts ✓ Clear
0004_auto.ts                   ✗ Unclear (rename this)
```

### Never Modify Applied Migrations

Once a migration has been applied anywhere (dev, staging, or production), treat it as immutable. The checksum validation will reject modified migrations.

If you made a mistake in an applied migration:
1. Create a new migration that fixes the mistake
2. Never edit the original file

## CI/CD Integration

### Check for Pending Migrations in CI

Add a step to verify no migrations are pending before deploying:

```yaml
# GitHub Actions example
- name: Check migration status
  run: |
    bun run migrate:status
    # Fail if any migrations are pending
```

### Deploy Migrations Before New Code

Always apply migrations before deploying new application code. New code may depend on the new schema:

```yaml
# Deployment pipeline
steps:
  - name: Deploy migrations
    run: bun run migrate:up

  - name: Deploy application
    run: # your deployment command
```

### Rollback Strategy

Keep your `down` functions accurate so you can roll back if needed:

```bash
# Roll back the latest migration
bun run migrate:down

# Roll back to a specific point
magnet migrate:down --to 0003_add_email_index
```

## Handling Dangerous Operations

### Column Drops

When you remove a field from your schema decorator, the generated migration will include `DROP COLUMN`. Before applying:

1. **Ensure no code references the column** — update application code first
2. **Consider a two-phase approach**:
   - Phase 1: Remove the column from application code, deploy
   - Phase 2: Run the DROP COLUMN migration separately

```typescript
// Phase 2 migration — safe to run after code is deployed
async up(db: MigrationDb) {
  // Application is already not writing to this column
  await db.execute(`ALTER TABLE "users" DROP COLUMN "legacy_field"`)
}
```

### Column Type Changes

Changing a column's type can fail with existing data. For safe type changes:

```typescript
async up(db: MigrationDb) {
  // 1. Add new column with the new type
  await db.execute(`ALTER TABLE "orders" ADD COLUMN "total_v2" NUMERIC(12,4)`)

  // 2. Copy and convert data
  await db.execute(`UPDATE "orders" SET "total_v2" = "total"::NUMERIC(12,4)`)

  // 3. Drop old column
  await db.execute(`ALTER TABLE "orders" DROP COLUMN "total"`)

  // 4. Rename new column
  await db.execute(`ALTER TABLE "orders" RENAME COLUMN "total_v2" TO "total"`)
}
```

## Backup Before Migrations

Always back up your production database before applying migrations, especially when they include dangerous operations:

```bash
# PostgreSQL backup example
pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql

# Then apply migration
bun run migrate:up
```

## Lock Timeout

If a server crashes during migration, it may leave a lock in the database. The lock expires after `lockTimeout` milliseconds (default: 60 seconds). Subsequent migration runs will automatically clear expired locks.

If you need to clear a lock immediately:

```sql
-- Manually clear the lock table
DELETE FROM "_magnet_migrations_lock";
```

## SQLite Limitations

SQLite has restrictions on `ALTER TABLE`:
- Cannot drop columns before SQLite 3.35.0
- Cannot change column types

For these operations with SQLite, use the "recreate table" pattern:

```typescript
async up(db: MigrationDb) {
  // 1. Create new table with desired schema
  await db.execute(`
    CREATE TABLE "users_new" (
      "id" TEXT PRIMARY KEY,
      "email" TEXT NOT NULL
      -- omit the dropped column
    )
  `)

  // 2. Copy data
  await db.execute(`INSERT INTO "users_new" SELECT "id", "email" FROM "users"`)

  // 3. Drop old table
  await db.execute(`DROP TABLE "users"`)

  // 4. Rename new table
  await db.execute(`ALTER TABLE "users_new" RENAME TO "users"`)
}
```

## Migration Reset (Development Only)

`magnet migrate:reset` and `magnet migrate:fresh` are for development only:

- **`migrate:reset`**: Rolls back all migrations, then re-applies them. Use when your down migrations aren't working correctly.
- **`migrate:fresh`**: Drops all tables, then re-applies all migrations. Use for a completely clean slate.

These commands require `--force` or interactive confirmation to prevent accidental use.

**Never use `migrate:reset` or `migrate:fresh` in production** — they will destroy all data.
