title: Notifications Module
description: Multi-channel notification system for in-app and external delivery
---

The Notifications Module provides a unified, pluggable system for sending notifications across multiple delivery channels. It powers the admin UI notification drawer and supports any number of external channels via the adapter pattern.

## Overview

The Notifications Module provides:

- **Platform channel**: Always-on in-app notifications persisted to the database and displayed in the admin drawer
- **Pluggable external channels**: Email and other channels via registered adapters
- **Bulk delivery**: Notify one user or many in a single call
- **Read/unread tracking**: Mark individual or all notifications as read
- **Automatic retention cleanup**: Remove old records based on a configurable period
- **Settings integration**: Enable/disable globally or per-channel from the admin Settings panel

## Architecture

```
NotificationService.notify(dto)
         │
         ├── platform channel (built-in)  ──→  database  ──→  admin drawer
         │
         └── external channels (adapters) ──→  email / SMS / webhook / ...
```

Any NestJS service in core, a plugin, or an application can inject `NotificationService` and call `notify()`. The service fans out to each requested channel, never throwing — failures are logged and the caller is unaffected.

## Basic Usage

### Inject the service

```typescript
import { Injectable } from '@nestjs/common'
import { NotificationService } from '@magnet-cms/core'

@Injectable()
export class MyService {
  constructor(private readonly notifications: NotificationService) {}

  async onSomethingHappened(userId: string) {
    await this.notifications.notify({
      userId,
      type: 'system',
      title: 'Action Required',
      message: 'Please review the pending changes.',
      href: '/content/pending',
    })
  }
}
```

Channels default to `['platform']` when omitted. To add email:

```typescript
await this.notifications.notify({
  userId,
  channels: ['platform', 'email'],
  type: 'content.published',
  title: 'Article Published',
  message: 'Your article "Hello World" has been published.',
  href: '/content/posts/hello-world',
})
```

### Bulk delivery

Pass an array of user IDs to notify multiple users in one call:

```typescript
await this.notifications.notify({
  userId: ['user-1', 'user-2', 'user-3'],
  type: 'system',
  title: 'Scheduled maintenance tonight',
  message: 'The system will be unavailable from 02:00–03:00 UTC.',
})
```

### Using from a plugin

```typescript
import { Injectable } from '@nestjs/common'
import { OnEvent, NotificationService } from '@magnet-cms/core'
import type { EventPayload } from '@magnet-cms/common'

@Injectable()
export class ContentPublishNotifier {
  constructor(private readonly notifications: NotificationService) {}

  @OnEvent('content.published', { async: true })
  async onContentPublished(
    payload: EventPayload<'content.published'>,
  ): Promise<void> {
    if (!payload.userId) return

    await this.notifications.notify({
      userId: payload.userId,
      type: 'content.published',
      title: 'Content Published',
      message: `Document "${payload.documentId}" in "${payload.schema}" is now live.`,
      href: `/content/${payload.schema}/${payload.documentId}`,
      metadata: { schema: payload.schema, documentId: payload.documentId },
    })
  }
}
```

## NotifyDto Reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `userId` | `string \| string[]` | Yes | Target user(s) |
| `type` | `string` | Yes | Notification category (free-form, used for icon mapping in UI) |
| `title` | `string` | Yes | Short title shown in the notification list |
| `message` | `string` | Yes | Full body / description |
| `channels` | `NotificationChannel[]` | No | Defaults to `['platform']` |
| `href` | `string` | No | Deep-link within the admin on click |
| `metadata` | `Record<string, unknown>` | No | Additional context stored with the record |

## Channels

### Platform channel (built-in)

The platform channel is always available — no configuration required. It:

- Persists notification records to the database under the user's ID
- Surfaces them in the admin UI notification drawer
- Supports read/unread state and pagination
- Is controlled by the `platformChannelEnabled` setting

### Email channel (adapter-based)

Email delivery requires registering an adapter. Enable the channel in Settings and pass an adapter when registering the module:

```typescript
// app.module.ts
import { MagnetModule } from '@magnet-cms/core'
import { MyEmailAdapter } from './my-email-adapter'

@Module({
  imports: [
    MagnetModule.forRoot({
      // ...
    }),
  ],
})
export class AppModule {}
```

<Callout type="info">
  `NotificationModule` is registered automatically inside `MagnetModule.forRoot()`. To supply channel adapters, pass them when bootstrapping using the adapter registration API described in [Custom Channel Adapters](#custom-channel-adapters).
</Callout>

### Custom channel adapters

Implement `NotificationChannelAdapter` from `@magnet-cms/common`:

```typescript
import type {
  NotificationChannelAdapter,
  NotificationChannelResult,
  NotificationSendPayload,
  NotificationRecipient,
} from '@magnet-cms/common'

export class MyEmailAdapter implements NotificationChannelAdapter {
  readonly channel = 'email' as const

  async send(
    payload: NotificationSendPayload,
    recipient: NotificationRecipient,
  ): Promise<NotificationChannelResult> {
    if (!recipient.email) {
      return { channel: 'email', success: false, error: 'No email address' }
    }

    try {
      await this.mailer.send({
        to: recipient.email,
        subject: payload.title,
        text: payload.message,
      })
      return { channel: 'email', success: true }
    } catch (error) {
      return {
        channel: 'email',
        success: false,
        error: error instanceof Error ? error.message : 'Unknown error',
      }
    }
  }
}
```

Register the adapter programmatically via `NotificationService.registerAdapter()`:

```typescript
import { Injectable, OnModuleInit } from '@nestjs/common'
import { NotificationService } from '@magnet-cms/core'
import { MyEmailAdapter } from './my-email-adapter'

@Injectable()
export class NotificationBootstrap implements OnModuleInit {
  constructor(private readonly notifications: NotificationService) {}

  onModuleInit() {
    this.notifications.registerAdapter(new MyEmailAdapter())
  }
}
```

## API Endpoints

All endpoints require JWT authentication (`Authorization: Bearer <token>`).

### List notifications

```http
GET /notifications
```

Returns paginated notifications for the current user, newest first.

**Query parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `limit` | `number` | Max results (default `20`) |
| `offset` | `number` | Skip N results for pagination |
| `unreadOnly` | `true` | Only return unread notifications |

**Response:**

```json
{
  "items": [
    {
      "id": "notif_abc123",
      "userId": "user_123",
      "type": "content.published",
      "title": "Article Published",
      "message": "Your article has been published.",
      "read": false,
      "href": "/content/posts/hello-world",
      "channels": ["platform"],
      "createdAt": "2026-03-14T10:30:00Z"
    }
  ],
  "total": 12,
  "unreadCount": 3,
  "limit": 20,
  "offset": 0
}
```

### Get unread count

```http
GET /notifications/unread-count
```

**Response:**

```json
{ "count": 3 }
```

### Mark one as read

```http
PATCH /notifications/:id/read
```

Returns `204 No Content`. No-ops if the notification belongs to a different user.

### Mark all as read

```http
PATCH /notifications/read-all
```

Marks every notification for the current user as read. Returns `204 No Content`.

### Send a notification (admin)

```http
POST /notifications
```

Sends a notification to one or more users. Restricted to admin users.

**Body:**

```json
{
  "userId": "user_123",
  "channels": ["platform", "email"],
  "type": "system",
  "title": "Scheduled Maintenance",
  "message": "The system will be down for maintenance on Saturday.",
  "href": "/announcements/maintenance"
}
```

`userId` can also be an array of strings for bulk delivery.

**Response:** `202 Accepted`

```json
{ "sent": true }
```

### Cleanup old notifications

```http
DELETE /notifications/cleanup?retentionDays=90
```

Deletes notifications older than `retentionDays` (defaults to the configured setting). Returns the number of deleted records.

```json
{ "deleted": 47 }
```

## Service API

Inject `NotificationService` for programmatic access:

```typescript
import { Injectable } from '@nestjs/common'
import { NotificationService } from '@magnet-cms/core'

@Injectable()
export class DashboardService {
  constructor(private readonly notifications: NotificationService) {}

  async getUserNotifications(userId: string) {
    // Paginated list with unread count
    return this.notifications.getForUser(userId, {
      limit: 20,
      offset: 0,
      unreadOnly: false,
    })
  }

  async getUnreadBadgeCount(userId: string) {
    return this.notifications.getUnreadCount(userId)
  }

  async dismissNotification(id: string, userId: string) {
    await this.notifications.markAsRead(id, userId)
  }

  async clearAll(userId: string) {
    await this.notifications.markAllAsRead(userId)
  }
}
```

## Settings

Configure notification behavior in the Admin UI under **Settings → Notifications**, or read settings programmatically:

```typescript
import { Injectable } from '@nestjs/common'
import { SettingsService, NotificationSettings } from '@magnet-cms/core'

@Injectable()
export class MyService {
  constructor(private readonly settings: SettingsService) {}

  async checkNotificationsEnabled() {
    const config = await this.settings.get(NotificationSettings)
    return config.enabled && config.platformChannelEnabled
  }
}
```

### Available settings

| Setting | Default | Description |
|---------|---------|-------------|
| `enabled` | `true` | Master switch — disables all notification delivery when `false` |
| `platformChannelEnabled` | `true` | Persist notifications to the database for in-app display |
| `emailChannelEnabled` | `false` | Deliver via email (requires a registered email adapter) |
| `retentionDays` | `90` | Days before old notification records are eligible for cleanup |

## Admin UI

The notification bell icon in the admin header displays a live unread badge and opens the notifications drawer. The drawer:

- Fetches the 20 most recent notifications via `GET /notifications`
- Polls the unread count every 60 seconds and on window focus
- Marks a notification as read when clicked
- Provides a "Mark all read" button
- Shows a loading spinner during the initial fetch

No configuration is required — the drawer is wired automatically once `NotificationModule` is active.

## Events

The `notification.created` event is added to the `EventName` union, making it available to `@OnEvent` handlers:

```typescript
import { Injectable } from '@nestjs/common'
import { OnEvent } from '@magnet-cms/core'
import type { EventPayload } from '@magnet-cms/common'

@Injectable()
export class NotificationAnalytics {
  @OnEvent('notification.created')
  async onNotificationCreated(
    payload: EventPayload<'notification.created'>,
  ): Promise<void> {
    // payload: { notificationId, targetUserId, type, channels }
    console.log(`Notification sent to ${payload.targetUserId}`)
  }
}
```
