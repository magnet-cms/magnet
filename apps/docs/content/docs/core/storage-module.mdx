## Overview

Magnet registers `StorageModule` when you bootstrap the app. To persist uploads to object storage, add a **storage** `MagnetProvider` to `MagnetModule.forRoot([...])` alongside your database adapter.

Without a storage provider, features that expect external storage may be limited—choose an adapter that matches your stack.

## Official adapters

| Adapter | Package | Use case |
|---------|---------|----------|
| Supabase Storage | `@magnet-cms/adapter-storage-supabase` | Supabase-hosted buckets |
| Amazon S3 | `@magnet-cms/adapter-storage-s3` | AWS S3 or S3-compatible APIs |
| Cloudflare R2 | `@magnet-cms/adapter-storage-r2` | R2 with public URL |

## Example (S3)

```typescript
import { Module } from '@nestjs/common'
import { MagnetModule } from '@magnet-cms/core'
import { MongooseDatabaseAdapter } from '@magnet-cms/adapter-db-mongoose'
import { S3StorageAdapter } from '@magnet-cms/adapter-storage-s3'

@Module({
  imports: [
    MagnetModule.forRoot([
      MongooseDatabaseAdapter.forRoot(),
      S3StorageAdapter.forRoot(),
    ]),
  ],
})
export class AppModule {}
```

## Injecting StorageService

`StorageModule` is registered globally — inject `StorageService` anywhere without importing the module:

```typescript
import { Injectable } from '@nestjs/common'
import { StorageService } from '@magnet-cms/core'

@Injectable()
export class DocumentService {
  constructor(private readonly storage: StorageService) {}

  async uploadUserDocument(
    file: Buffer,
    filename: string,
    userId: string,
  ): Promise<string> {
    const media = await this.storage.upload(file, filename, {
      folder: 'documents',
      createdBy: userId,
    })
    return media.url
  }

  async uploadManyFiles(
    files: Array<{ buffer: Buffer; filename: string }>,
    userId: string,
  ) {
    return this.storage.uploadMany(
      files.map(f => ({
        file: f.buffer,
        filename: f.filename,
        options: { folder: 'bulk', createdBy: userId },
      })),
    )
  }

  async deleteFile(mediaId: string): Promise<boolean> {
    return this.storage.delete(mediaId)
  }
}
```

### StorageService API

| Method | Description |
|--------|-------------|
| `upload(file, filename, options?)` | Upload a single file; returns `Media` record |
| `uploadMany(items)` | Upload multiple files in one call |
| `uploadChunked(...)` | Upload large files in chunks |
| `delete(mediaId)` | Delete a file by media record ID |
| `deleteMany(mediaIds)` | Delete multiple files |
| `createFolder(name, path, options?)` | Create a media folder |
| `listMedia(options?)` | List media with pagination and filters |

## PII encryption (encrypted folders)

Magnet supports **AES-256-GCM encryption at rest** for sensitive files (PII, documents, health data). Each user gets a unique 256-bit key stored automatically in the [Vault module](/docs/core/vault-module).

### How it works

1. **Vault key auto-created** — on first encrypted upload for a user, `MediaEncryptionService` generates a random 256-bit key and stores it in the vault at `media-key/{userId}`
2. **Folder auto-created** — if `autoCreatePrivateFolders` is enabled (default: `true`), Magnet creates a `private/{userId}` folder automatically when no folder is specified
3. **File encrypted before upload** — the file buffer is encrypted with AES-256-GCM; the IV and auth tag are stored alongside the media record in the database
4. **Transparent decryption** — on download, Magnet retrieves the key from the vault, decrypts the buffer, and streams the plaintext to the client

### Enabling encryption

Enable in the admin panel under **Settings → Media → Security**:

| Setting | Default | Description |
|---------|---------|-------------|
| Enable File Encryption | `false` | Master toggle — requires a vault adapter |
| Auto-create Private Folders | `true` | Creates `private/{userId}` on first encrypted upload |

> **Requires:** A vault adapter configured in `MagnetModule.forRoot()`. See [Vault module](/docs/core/vault-module).

### Uploading an encrypted file

Pass `encrypt: true` and `ownerId` to the `upload()` call:

```typescript
import { Injectable } from '@nestjs/common'
import { StorageService } from '@magnet-cms/core'

@Injectable()
export class PatientDocumentService {
  constructor(private readonly storage: StorageService) {}

  async uploadSensitiveDocument(
    fileBuffer: Buffer,
    filename: string,
    patientUserId: string,
  ) {
    // Magnet will:
    //  1. Look up (or create) the vault key for patientUserId at media-key/{patientUserId}
    //  2. Create the folder private/{patientUserId} if it doesn't exist
    //  3. Encrypt the buffer with AES-256-GCM
    //  4. Upload the encrypted bytes to your storage adapter
    //  5. Store IV + authTag in the Media record for decryption
    const media = await this.storage.upload(fileBuffer, filename, {
      encrypt: true,
      ownerId: patientUserId,
      // folder is auto-set to private/{patientUserId} when omitted
    })

    return {
      mediaId: media.id,
      url: media.url,        // URL points to the encrypted bytes
      isEncrypted: true,
    }
  }
}
```

### Retrieving encrypted files

Encrypted files are decrypted transparently when accessed through Magnet's media endpoints. The vault key is fetched per request, the buffer is decrypted in memory, and plaintext bytes are returned to the authorized requester.

> Only the file owner (matching `ownerId`) or an admin can access encrypted files. Unauthorized requests receive `403 Forbidden`.

## Storage settings

Configure via admin panel under **Settings → Media**:

| Setting | Default | Description |
|---------|---------|-------------|
| Max File Size | `10` MB | Maximum upload size |
| Allowed MIME Types | _(all)_ | Comma-separated MIME type whitelist |
| Enable File Encryption | `false` | AES-256-GCM at-rest encryption |
| Auto-create Private Folders | `true` | Folder auto-created for encrypted uploads |
